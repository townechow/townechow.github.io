d:T3775,<h1>HTTP 资源标识</h1>
<h2>统一资源标识符 URI</h2>
<p>统一资源标志符（Uniform Resource Identifier，简称 URI）是一个用于标识（区分）互联网资源名称的字符串。该种标识允许用户对网络种的资源通过特定的协议进行交互操作。</p>
<p>URI 可以进一步分为 <a href="/front-end/computer-networks/http/#%E7%BB%9F%E4%B8%80%E8%B5%84%E6%BA%90%E5%AE%9A%E4%BD%8D%E7%AC%A6-url">URL</a> 和 <a href="/front-end/computer-networks/http/#%E6%B0%B8%E4%B9%85%E7%BB%9F%E4%B8%80%E8%B5%84%E6%BA%90%E5%AE%9A%E4%BD%8D%E7%AC%A6-urn">URN</a>。URI 是以一种抽象的，高层次概念定义统一资源标识，而 URL 和 URN 则是具体的资源标识的方式。</p>
<p>通用的格式：</p>
<pre class="hljs"><code>[scheme][://]user:passwd@[host:port][path][?query][#fragment]
</code></pre>
<h3>编码方式</h3>
<p>URI 只能使用 ASCII，ASCII 之外的字符是不支持显示的，而且还有一部分符号是界定符，如果不加以处理就会导致解析出错。</p>
<p>因此，URI 引入了 <strong>编码</strong> 机制，将所有 <strong>非 ASCII 码字符</strong> 和 <strong>界定符</strong> 转为十六进制字节值，然后在前面加个 <code>%</code>。</p>
<p>如：空格被转义成了 <code>%20</code>，<strong>三元</strong> 被转义成了 <code>%E4%B8%89%E5%85%83</code>。</p>
<h3>方案或协议</h3>
<p><code>http://</code> 告诉浏览器使用何种协议。对于大部分 Web 资源，通常使用 HTTP 协议或其安全版本 HTTPS 协议。</p>
<p>另外，浏览器也知道如何处理其他协议。例如，</p>
<ul>
<li><code>mailto</code>：协议指示浏览器打开邮件客户端</li>
<li><code>ftp</code>：协议指示浏览器处理文件传输。</li>
</ul>

<h3>主机</h3>
<img alt="HTTP 主机" src="../../assets/http/http-url-domain.png" width="360" /><p>上述例子中 <code>www.example.com</code> 既是一个域名，也代表管理该域名的机构。它指示了需要向网络上的哪一台主机发起请求。当然，也可以直接向主机的 IP 地址发起请求。但直接使用 IP 地址的场景并不常见。</p>
<h3>端口</h3>
<img alt="HTTP 端口" src="../../assets/http/http-url-port.png" width="360" /><p><code>:80</code> 是端口。它表示用于访问 Web 服务器上资源的技术 <strong>门</strong>。如果访问的该 Web 服务器使用 HTTP 协议的标准端口（HTTP 为 80，HTTPS 为 443）授予对其资源的访问权限，则通常省略此部分。否则端口就是 URI 必须的部分。</p>
<h3>路径</h3>
<img alt="HTTP 路径" src="../../assets/http/http-url-path.png" width="360" /><p><code>/path/to/myfile.html</code> 是 Web 服务器上资源的路径。</p>
<p>在 Web 的早期，类似这样的路径表示 Web 服务器上的物理文件位置。现在，它主要是由没有任何物理实体的 Web 服务器抽象处理而成的。</p>
<h3>查询</h3>
<img alt="HTTP 查询" src="../../assets/http/http-url-parameters.png" width="360" /><p><code>?key1=value1&amp;key2=value2</code> 是提供给 Web 服务器的额外参数。这些参数是用 <code>&amp;</code> 符号分隔的键/值对列表。</p>
<p>Web 服务器可以在将资源返回给用户之前使用这些参数来执行额外的操作。每个 Web 服务器都有自己的参数规则，想知道特定 Web 服务器如何处理参数的唯一可靠方法是询问该 Web 服务器所有者。</p>
<h3>片段</h3>
<img alt="HTTP 查询" src="../../assets/http/http-url-anchor.png" width="360" /><p><code>#SomewhereInTheDocument</code> 是资源本身的某一部分的一个锚点。锚点代表资源内的一种 <strong>书签</strong>，它给予浏览器显示位于该 <strong>加书签</strong> 点的内容的指示。</p>
<p>例如，在 HTML 文档上，浏览器将滚动到定义锚点的那个点上；在视频或音频文档上，浏览器将转到锚点代表的那个时间。</p>
<p>值得注意的是 <code>#</code> 号后面的部分，也称为 <strong>片段标识符</strong>，永远不会与请求一起发送到服务器。</p>
<h2>统一资源定位符 URL</h2>
<p>统一资源定位符（Uniform Resource Locator，简称 URL）是 URI 最常见的形式，有时候也被俗称为网页地址（网址），如同是网络上的门牌，是因特网上标准的资源的地址。</p>
<pre class="hljs"><code>https://developer.mozilla.org
https://developer.mozilla.org/en-US/docs/Learn/
https://developer.mozilla.org/en-US/search?q=URL
</code></pre>
<p>在浏览器的地址栏中输入上述任一地址，浏览器就会加载相应的网页（资源）。</p>
<p>URL 由多个必须或可选的组件构成。下面给出了一个复杂的 URL：</p>
<pre class="hljs"><code>http://www.example.com:80/path/to/myfile.html?key1=value1&amp;key2=value2#SomewhereInTheDocument
</code></pre>
<h2>永久统一资源定位符 URN</h2>
<p>永久统一资源定位符（Uniform Resource Name，简称 URN）是另一种形式的 URI，它通过特定命名空间中的唯一名称来标识资源。</p>
<pre class="hljs language-http"  style=--lang:"http" ><code>urn:isbn:9780141036144
urn:ietf:rfc:7230
</code></pre>
<h2>Data URI Scheme</h2>
<p>Data URI Scheme，即前缀为 <code>data:</code> 协议的 URI，其允许内容创建者向文档中嵌入小文件。</p>
<pre class="hljs"><code>data:[&lt;mediatype&gt;][;charset=&lt;charset&gt;][;&lt;encoding&gt;],&lt;encoded-data&gt;
</code></pre>
<p>Data URI Scheme 由四个部分组成：</p>
<ul>
<li>协议头 <code>data:</code>：标识这个内容为 Data URI Scheme 资源</li>
<li>MIME 类型（可选项）：查看下方详细描述</li>
<li>源文本的字符集编码方式 <code>[;charset=&lt;charset&gt;]</code>：默认编码是 <code>charset=US-ASCII</code>，即数据部分的每个字符都会自动编码为 <code>%xx</code></li>
<li>数据编码方式 <code>[;&lt;encoding&gt;]</code>：默认 <code>US-ASCII</code> 和 <code>BASE64</code> 两种</li>
<li>编码后的数据 <code>&lt;encoded data&gt;</code></li>
</ul>
<p><strong>使用 Data URI Scheme 的优劣势</strong></p>
<ul>
<li>优势
<ul>
<li>减少 HTTP 请求</li>
<li>当访问外部资源很麻烦或受限时（例如资源服务器 IP 被禁用）</li>
<li>当图片是在服务端用程序动态生成，每个访问用户显示均不同</li>
<li>当图片的体积比较小，占用 HTTP 会话性价比不高</li>
<li>没有图片更新要重新上传，还要清理缓存的问题</li>
</ul>
</li>
<li>劣势
<ul>
<li>Base64 编码的数据体积通常为原数据的 4/3，也就是 Data URI Scheme 形式的图片会比二进制格式的图片体积大 1/3</li>
<li>Data URI Scheme 形式的图片不会被浏览器缓存，意味着每次访问页面都需要被重新加载</li>
<li>不适合用于懒加载中</li>
<li>移动端性能优先并不适宜用 Data URI Scheme 技术，解码耗费 CPU</li>
</ul>
</li>
</ul>

<h2>MIME 类型</h2>
<p>媒体类型（Multipurpose Internet Mail Extensions，简称 MIME 类型）是一种标准，用于表示文档、文件或字节流的性质和格式。在 HTTP 中，HTTP 会从 MIME 类型中取部分标记报文 Body 部分的数据类型，这些类型体现在 <code>Content-Type</code> 这个字段，当然这是针对于发送端而言的，接收端想要收到特定类型的数据，也可以使用 <code>Accept</code> 字段。</p>
<p>浏览器通常使用 MIME 类型（而不是文件扩展名）来确定如何处理文档。因此服务器设置正确以将正确的 MIME 类型附加到响应对象的头部是非常重要的。</p>
<p>MIME 类型对大小写不敏感，但是传统写法都是小写。</p>

<h3>复合类型</h3>
<pre class="hljs"><code>multipart/form-data
multipart/byteranges
</code></pre>
<p>Multipart 类型表示细分领域的文件类型的种类，经常对于不同的 MIME 类型。这是复合文件的一种表现方式。</p>
<ul>
<li><code>multipart/form-data</code>：用于联系 HTML Forms 和 POST 方法</li>
<li><code>multipart/byteranges</code>：使用状态码 <code>206 Partial Content</code> 来发送整个文件的子集，而 HTTP 对不能处理的复合文件使用特殊的方式，将信息直接传送给浏览器</li>
</ul>
<p><code>multipart/form-data</code> 可用于 HTML 表单从浏览器发送信息给服务器。作为多部分文档格式，它由边界线（由 <code>--</code> 开始的字符串）划分出的不同部分组成。每个部分有自己的实体，以及自己的 HTTP 请求头，<code>Content-Disposition</code> 和 <code>Content-Type</code> 用于文件上传领域。</p>
<h3>MIME 嗅探</h3>
<p>在却是 MIME 类型或客户端认为文件设置了错误的 MIME 类型时，浏览器可能会通过查看资源来进行 MIME 嗅探。每个浏览器在不同的情况下会执行不同的操作。因为这个操作会有一些安全问题，有的 MIME 类型表示可执行内容而有些是不可执行内容。浏览器可以通过请求头 <code>Content-Type</code> 来设置 <code>X-Content-Type-Options</code> 以阻止 MIME 嗅探。</p>
