d:T60ca,<h1>HTTPS</h1>
<p><strong>超文本传输安全协议</strong>（Hyper Text Transfer Protocol over Secure Socket Layer，HTTPS），是一种通过计算机网络进行安全通信的传输协议。HTTPS 经由 HTTP 进行通信，但利用 &lt;span style=&quot;font-weight:bold;color:red&quot;&gt;SSL / TLS&lt;/span&gt; 来加密数据包。HTTPS 开发的主要目的，是提供对网站服务器的身份认证，保护交换数据的隐私与完整性。</p>
<pre class="hljs language-js"  style=--lang:"js" ><code><span class="hljs-variable constant_">HTTPS</span> = <span class="hljs-variable constant_">HTTP</span> + <span class="hljs-variable constant_">SSL</span>（加密 + 认证 + 完整性保护）
</code></pre>
<p>HTTP 协议采用明文传输信息，存在信息窃听、信息篡改和信息劫持的风险，而协议 SSL / TLS 具有<strong>身份验证</strong>、<strong>信息加密</strong>和<strong>完整性校验</strong>的功能，可以避免此类问题发生。</p>
<p>SSL / TLS 全称安全传输层协议 Transport Layer Security，是介于 TCP 和 HTTP 之间的一层安全协议，不影响原有的 TCP 协议和 HTTP 协议，所以使用 HTTPS 基本上不需要对 HTTP 页面进行太多的改造。</p>
<h2>安全问题</h2>
<p>使用 HTTP 协议通信存在的问题：</p>
<ul>
<li><strong>窃听</strong>：通信内容不经加密有可能&lt;span style=&quot;font-weight:bold;color:red&quot;&gt;被第三方捕获&lt;/span&gt;</li>
<li><strong>伪造</strong>：第三方拦截通信内容，并&lt;span style=&quot;font-weight:bold;color:red&quot;&gt;伪装成发送方&lt;/span&gt;向接收方发送通信内容</li>
<li><strong>否认</strong>：发送方将&lt;span style=&quot;font-weight:bold;color:red&quot;&gt;恶意数据&lt;/span&gt;发送给接受方并否认这些数据是它发送的</li>
</ul>
<p>窃听属于数据加密问题，而伪造和否认则属于身份认证问题。</p>
<h3>窃听</h3>
<img alt="窃听" src="../../assets/https/eavesdropping.jpg" width="720" /><h3>伪造</h3>
<img alt="伪造" src="../../assets/https/forge.jpg" width="720" /><h3>否认</h3>
<img alt="否认" src="../../assets/https/deny.png" width="720" /><h2>解决方案</h2>
<p>对通信内容进行加密的解决方案：</p>
<ul>
<li><strong>对称加密</strong>：使用相同密钥加密/解密，但是钥匙交付问题无法解决</li>
<li><strong>非对称加密</strong>：公钥加密数据，私钥解密数据，但是加密/解密消耗时间较多</li>
<li><strong>混合加密</strong>：公钥加密密钥，密钥加密数据，私钥解密密钥，密钥解密数据（非对称传送密钥，密钥传送数据）。解决非对称加密效率问题</li>
<li><strong>中间人攻击</strong>：秘密替换公钥窃取数据，源于服务端无法确认客户端公钥是否合法</li>
<li><strong>消息鉴别码</strong>：<code>MAC</code> 密钥和密文组成的字符串的哈希值。能够解决 <code>伪造</code> 问题，但无法解决 <code>否认</code> 问题</li>
<li><strong>数字签名</strong>：服务端创建数字签名，而客户端只验证签名是否争取。解决 <code>否认</code> 问题</li>
<li><strong>数字证书</strong>：由权威机构背书创建证书，保证公钥不被篡改。解决 <code>中间人攻击</code> 问题</li>
</ul>
<h3>对称加密</h3>
<p><strong>对称加密算</strong>法（Symmetric Key Algorithms）加密使用的密钥和解密使用的密钥是相同的。也就是说，加密和解密都是使用的同一个密钥。因此对称加密算法要保证安全性的话，密钥要做好保密，只能让使用的人知道，不能对外公开。</p>
<img alt="对称加密" src="../../assets/https/symmetric-encryption.jpg" width="720" /><p><strong>对称密钥加密</strong>一个很重要的特点就是使用相同的密钥进行加密和解密</p>
<img alt="对称密钥交付问题" src="../../assets/https/symmetric-encryption-problem.jpg" width="720" /><p>假设 B 是没有解密钥匙，所以 A 需要通过互联网将钥匙发送给 B。</p>
<p>但是 X 也有可能看到并获取到这个钥匙，因此，X 也可以通过这个钥匙来解密密文。</p>
<p>上面这个场景就会引出一个新问题，这个问题被称为<strong>钥匙交付问题</strong>，那怎么解决这个问题？</p>
<h3>非对称加密</h3>
<p>为了解决上面的<strong>钥匙交付问题</strong>，我们这里引入一个新的方法 —— <strong>非对称加密</strong>（Asymmetric Key Algorithms）</p>
<ul>
<li>用于加密的密钥被称为<strong>公钥</strong>，用于解密的密钥被称为<strong>私钥</strong></li>
<li>跟<strong>对称加密</strong>相比，公开密钥加密往往<strong>需要更多的时间</strong>用于加密和解密</li>
</ul>
<p><strong>非对称加密过程</strong></p>
<p>接收方 B 创建一个公钥和一个私钥，公钥被发送给 A。</p>
<img alt="非对称加密过程-创建" src="../../assets/https/asymmetric-encryption-creation.jpg" width="720" /><ul>
<li>A 使用从 B 收到的公钥加密数据，将密文发送给 B</li>
<li>B 使用私钥解密从 A 接收到的密文，得到原始数据</li>
</ul>
<img alt="非对称加密过程-传输" src="../../assets/https/asymmetric-encryption-transport.jpg" width="720" /><p>在这个过程中：</p>
<ul>
<li>密文和公钥也可能被恶意第三方 X 截获</li>
<li>但是私钥是 B 保存的，X 无法获取到，自然没有办法解密密文</li>
<li>这样就很好的解决了<strong>钥匙交付问题</strong></li>
</ul>
<p><strong>公开密钥加密的问题</strong></p>
<ol>
<li>加密和解密都需要耗费时间，有一种叫<strong>混合加密</strong>的方法可以解决这个问题</li>
<li>公开密钥的可靠性</li>
</ol>
<h3>混合加密</h3>
<img alt="混合加密" src="../../assets/https/mixed-encryption.jpg" width="720" /><p>混合密钥加密分为两个步骤：</p>
<ul>
<li>通过公开密钥加密传递密钥</li>
<li>通过速度更快的对称密钥加密方法传递数据</li>
</ul>
<h3>中间人攻击</h3>
<p>中间人攻击（Man In the Middle Attack）。</p>
<p>为了更好地理解公开密钥加密的可靠性问题，我们回到传递公钥的场景。</p>
<img alt="中间人攻击" src="../../assets/https/man-in-the-middle-attack.jpg" width="720" /><p>A 拿到的其实是 X 发送给他的伪造公钥，但是 A 无法察觉。</p>
<img alt="中间人攻击2" src="../../assets/https/man-in-the-middle-attack-2.jpg" width="720" /><p>最后，X 用他自己的密钥加密响应数据，并发送给 A，就这样，虽然 A、B 双方能顺利完成通信，但是恶意的第三方 X 能看到解密后的请求数据和响应数据，而 A、B 双方则毫不知情。</p>
<p>中间人攻击的攻击方式：</p>
<ul>
<li>域名污染：DNS 解析的中间链点返回中间人错误 IP</li>
<li>ARP 欺骗</li>
</ul>
<p>潜在的中间人攻击威胁：</p>
<ul>
<li>浏览器弹出的证书警告，可能访问的是钓鱼网站或假冒的服务器</li>
<li>公共 Wifi 下，没有 HTTPS 加密的网站不要随便登录</li>
<li>在任何网站登录账号必须走 HTTPS 加密协议</li>
</ul>
<h3>消息鉴别码</h3>
<p>消息鉴别码（Message Authentication Code，简写 MAC）是一种用于确认消息完整性（数据不会被篡改）和可靠性（消息认证是确认消息来自正确的发送者）的技术。消息鉴别码可以理解为<strong>密钥</strong>和<strong>密文</strong>组成的字符串的<strong>哈希值</strong>。</p>
<img alt="消息鉴别码" src="../../assets/https/mac.jpg" width="720" /><p>消息鉴别码虽然可以解决&lt;span style=&quot;font-weight:bold;color:red&quot;&gt;伪造&lt;/span&gt;问题，但是仍然无法避免<strong>否认</strong>问题。</p>
<h3>数字签名</h3>
<p>为了解决这个&lt;span style=&quot;font-weight:bold;color:red&quot;&gt;否认&lt;/span&gt;问题，我们接下来看看<strong>数字签名</strong>方法。</p>
<p>数据发送方通过约定的哈希函数从数据报文中生成报文摘要（Digest），然后利用私钥对报文摘要进行加密，这个加密后的摘要将作为报文的<strong>数字签名</strong>和<strong>数据报文</strong>一起发送给接收方。</p>
<p>数据接收方首先使用与发送方一致的哈希函数从接收到的原始报文中计算出报文摘要，接着再用发送方的公钥对数字签名进行解密，得出摘要。如果摘要相同，那么接收方就能确认该数字签名是发送方的。</p>
<img alt="数字签名" src="../../assets/https/digital-sign.jpg" width="720" /><h3>数字证书</h3>
<p>虽然上面的方法已经能避免窃听、伪造、否认等问题，但是现在还是没办法避免<strong>中间人攻击</strong>，因为我们还是没办法验证公钥的所有者，因此我们需要&lt;span style=&quot;font-weight:bold;color:red&quot;&gt;数字证书&lt;/span&gt;系统来验证公钥的所有者。</p>
<p>接下来，先看看数字证书申请的过程，我们将数字证书认证机构（Certificate Authority）称之为 CA。</p>
<p><strong>数字证书申请过程：</strong></p>
<ol>
<li>
<p>服务器方向第三方机构 CA <strong>提交公钥</strong>、<strong>组织信息</strong>、<strong>个人信息（域名）</strong> 等信息并申请认证</p>
</li>
<li>
<p>CA 通过线上、线下等多种手段验证申请者提供信息的真实性，如组织是否存在、企业是否合法、是否拥有域名的所有权等</p>
</li>
<li>
<p>如信息审核通过，CA 会向申请者签发认证文件证书。证书包含以下信息：<strong>申请者公钥</strong>、<strong>申请者的组织</strong>和<strong>个人信息</strong>、<strong>签发机构 CA 的信息</strong>、<strong>有效时间</strong>、<strong>证书序列号</strong>等信息的明文，同时包含<strong>数字签名</strong>。</p>
</li>
</ol>
<p>数字签名通过<strong>散列函数</strong>计算公开的明文信息的<strong>信息摘要</strong>，然后，采用 CA 提供的私钥对信息摘要进行加密，加密后的信息摘要即为<strong>数字签名</strong></p>
<img alt="数字证书" src="../../assets/https/digital-certificate.jpg" width="720" /><p>现在 B 已经申请到一个数字证书了，那么怎么使用数字证书来检验公钥 PB 是属于 B 呢。</p>
<p><strong>数字证书校验流程：</strong></p>
<ol>
<li>
<p>客户端向服务端发送请求时，服务端返回数字证书文件</p>
</li>
<li>
<p>客户端读取证书中的相关的明文信息，采用与加密时相同的<strong>散列函数</strong>计算得到信息摘要，然后利用内置客户端中的受信任的 CA 公钥解密数字签名，对比证书的信息摘要</p>
<ul>
<li>如果一致，则可以确认证书的合法性，即公钥合法</li>
<li>如果不一致，则可以确认证书为非法，即公钥非法</li>
<li>如果 CA 不被信任，则找不到对应 CA 证书，证书也会被判定为非法，即公钥非法</li>
</ul>
</li>
<li>
<p>客户端验证证书相关的域名信息、有效时间等信息</p>
</li>
</ol>
<img alt="数字证书2" src="../../assets/https/digital-certificate-2.jpg" width="720" /><p>现在可以验证 PB 是属于 B 的，但是怎么验证 PC 是属于受信任的 CA 的呢。</p>
<img alt="数字证书3" src="../../assets/https/digital-certificate-3.jpg" width="720" /><p>事实上，认证机构形成一个树形结构，高级别的权威机构为较低级别的机构创建证书，那就是说，如果要验证的话，就是一级一级向上认证，信任链条的最终是 Root CA，他采用自签名，对他的签名是无条件的信任。</p>
<img alt="数字证书4" src="../../assets/https/digital-certificate-4.jpg" width="720" /><h2>工作流程</h2>
<p>HTTPS 在传输数据之前需要客户端（浏览器）与服务端之间进行一次握手，在握手过程中将确立双方加密传输数据的密码信息。TLS/SSL 协议使用<strong>非对称加密</strong>、<strong>对称加密</strong>以及 <strong>HASH 算法</strong>确保信息通信的安全。</p>
<img alt="HTTPS Workflow" src="../../assets/https/workflow.jpg" width="720" /><p><strong>具体流程：</strong></p>
<ol>
<li>
<p><strong>Client Hello</strong>：客户端发起 HTTPS 请求。</p>
<ul>
<li>不同的客户端对不同的加密算法的支持度不同，因此客户端会将支持的一套<strong>加密套件</strong>（Cipher Suite）发送给服务端（除了加密组合方式，报文中还包括 TLS 的协议版本、随机生成的密码串记为 Random1、会话 ID、域名 ServerName 以及签名算法）</li>
</ul>
</li>
<li>
<p><strong>Server Hello</strong>：服务端从客户端支持的加密规则中选用一组加密算法（包括用于生成通信密钥的算法），并随机生成密码串记为 Random2，后续将数据传回客户端。</p>
<ul>
<li><strong>Certificate</strong>：服务端收到 Client Hello 之后，向客户端发送 CA 认证的数字证书，用来鉴别服务端身份信息，同时还要生成一个 <strong>随机数</strong> 同时保存在服务端和发送给客户端。证书包含 <strong>网站信息</strong>、<strong>由 CA 私钥加密的通信公钥</strong>，以及 <strong>经证书的颁发机构的私钥加密产生的数字签名</strong> 等信息。</li>
<li><strong>Server Hello Done</strong>：表示服务端宣告第一阶段的客户端服务端握手协商结束</li>
<li>可选：<strong>Certificate Request</strong>：必要情况下，要求客户端发送证书验证身份</li>
<li>可选：<strong>Server Key Exchange</strong>：如果 CA 认证的数字证书提供的信息不够，服务端还可发送提供补充信息</li>
</ul>
</li>
<li>
<p><strong>Client Finish</strong></p>
<ul>
<li><strong>Client Key Exchange</strong>
<ul>
<li>客户端收到 CA 数字证书并通过验证，然后通过 CA 公钥解密获取到 <strong>服务端公钥</strong></li>
<li>客户端先读取数字证书中相关的明文信息，采用相同的 Hash 函数计算得到<strong>信息摘要</strong>，然后利用内置在客户端中的对应的 CA 的公钥解密<strong>数字签名</strong>数据，并与数字证书的摘要进行对比
<ul>
<li>如果一致，则证书合法，即公钥合法（如果证书受信任，则客户端地址栏会显示小锁头）</li>
<li>如果不一致，则证书非法，即公钥非法（如果证书不受信任，则会提示用户证书存在问题）</li>
</ul>
</li>
<li>如果证书受信任，客户端会根据两个 pubkey 随机数生成一串随机数的密钥记为 Pre-Master 随机数，并利用两次 Hello 报文产生的两个随机数 Random1 和 Random2，利用对称密钥生成算法生成对称密钥 <code>enc_key = Func(Random1, Random2, Pre-Master)</code>（即<strong>消息鉴别码</strong>：即用于终端数据交互的&lt;span style=&quot;font-weight:bold;color:red&quot;&gt;对称密钥&lt;/span&gt;），并用证书中提供的服务端公钥加密该随机数</li>
<li><strong>传送加密信息</strong>：使用约定好的 HASH 算法计算握手信息，并使用<strong>对称密钥</strong>对消息进行加密，最后将之前生成的所有信息发送给服务端</li>
</ul>
</li>
<li><strong>Change Cipher Spec</strong>：该报文通知服务端，此后的通信都将使用协商好的加密算法计算对称密钥进行加密通信（也就是使用两个随机数以及第三个 Pre-master key/secret 随机数一起算出一个对称密钥 <strong>session key/secret</strong>）</li>
<li><strong>Finished</strong> 报文：该报文包括连接至此的所有报文的校验值，使用服务端公钥进行加密</li>
<li>可选：<strong>ClientCertificate</strong> 报文：如果服务端请求，客户端需要发送 CA 数字证书</li>
<li>可选：<strong>CertificateVerify</strong> 报文：服务端如果要求 CA 数字证书，那么需要通过 HASH 算法计算一个服务端发送来的信息摘要</li>
</ul>
</li>
<li>
<p><strong>Server Finish</strong>：使用私钥将信息解密取出密钥，使用密钥解密客户端发来的握手信息，并验证 HASH 是否与浏览器发来的一致</p>
<ul>
<li><strong>New Session Ticket</strong>：服务端最后对客户端发送过来的 Finished 报文使用服务端私钥进行解密校验</li>
<li><strong>Change Cipher Spec</strong>：报文通知服务端，此后的通信都将使用协商好的加密算法计算对称密钥 session key/secret 进行加密通信</li>
<li><strong>Finished</strong> 报文：标志 TLS 连接建立成功</li>
</ul>
</li>
<li>
<p>浏览器解密并计算握手信息的 Hash，如果与服务端发来的 Hash 一致，此时握手过程结束，之后所有的通信数据将由之前客户端生成的对称密钥并利用堆成加密算法进行加密</p>
</li>
</ol>
<p>这里客户端与服务端互相发送加密的握手消息并验证，目的是为了保证双方都获得了一致的密码，并且可以正常的加密解密数据，为后续真正数据的传输做一次测试。</p>
<p>客户端取出提前内置在手机内部的认证机构的公钥</p>
<ol start="2">
<li>用认证机构的公钥去解密公钥证书里的数字签名 从而得到数字指纹</li>
<li>客户端对公钥证书的服务器公钥进行 数字摘要算法 从而生成数字指纹</li>
<li>对比客户端自己生成的数字指纹(第 3 步)和解密得到的数字指纹(第 2 步)是否一致 如果一致则公钥证书验证通过 就可以进行接下来的握手步骤了</li>
</ol>
<blockquote>
<ul>
<li>Client Hello 包含域名的信息是比较特别的，因为域名是工作在应用层 HTTP 里的，而握手是发生在 TLS 还在传输层。在传输层里面就把域名信息告诉服务端，好让服务器根据域名发送相应的证书。</li>
</ul>
</blockquote>
<h2>相关算法</h2>
<h3>散列函数</h3>
<p>常见的散列函数 Hash 算法：<a href="https://zh.wikipedia.org/wiki/MD5" target="_blank" rel="noopener noreferrer nofollow">MD5</a>、<a href="https://zh.wikipedia.org/wiki/SHA-1" target="_blank" rel="noopener noreferrer nofollow">SHA1</a>、<a href="https://zh.wikipedia.org/wiki/SHA%E5%AE%B6%E6%97%8F" target="_blank" rel="noopener noreferrer nofollow">SHA256</a></p>
<p>散列函数特点是函数单向不可逆转、对输入非常敏感、输出长度固定，针对数据的任何修改都会改变散列函数的结果，用于<strong>防止信息篡改</strong>并<strong>验证数据的完整性</strong>。</p>
<p>在信息传递过程中，散列函数不能单独实现信息信息防篡改，因为<strong>明文传输</strong>，中间人可以修改信息之后重新计算信息摘要，因此需要对传输的信息以及信息摘要进行加密。</p>
<h3>对称加密</h3>
<p>常见的对称加密算法：<a href="https://zh.wikipedia.org/wiki/AES%E6%8C%87%E4%BB%A4%E9%9B%86" target="_blank" rel="noopener noreferrer nofollow">AES-CBC</a>、<a href="https://zh.wikipedia.org/wiki/%E8%B3%87%E6%96%99%E5%8A%A0%E5%AF%86%E6%A8%99%E6%BA%96" target="_blank" rel="noopener noreferrer nofollow">DES</a>、<a href="https://zh.wikipedia.org/wiki/3DES" target="_blank" rel="noopener noreferrer nofollow">3DES</a>、<a href="https://zh.wikipedia.org/wiki/RC4" target="_blank" rel="noopener noreferrer nofollow">RC4</a></p>
<p>相同的密钥可以用于信息的加密和解密，掌握密钥才能获取信息，能够防止信息窃听，通信方式是 1 对 1。</p>
<p>对称加密的优势是信息传输 1 对 1，需要共享相同的密码，密码的安全是保证信息安全的基础，服务器和 N 个客户端通信，需要维持 N 个密码记录，且缺少修改密码的机制。</p>
<h3>非对称加密</h3>
<p>常见非对称加密算法：<a href="https://zh.wikipedia.org/wiki/RSA%E5%8A%A0%E5%AF%86%E6%BC%94%E7%AE%97%E6%B3%95" target="_blank" rel="noopener noreferrer nofollow">RSA</a>、DSA / DSS</p>
<p>密钥成对出现，一般称为公钥（公开）和私钥（保密），公钥加密的信息只能私钥解开，私钥加密的信息只能公钥解开。因此掌握公钥的不同客户端之间不能互相解密信息，只能和掌握私钥的服务器进行加密通信，服务器可以实现 1 对 N 的通信，客户端也可以用来验证掌握私钥的服务器身份。</p>
<p>非对称加密的特点是 1 对多，服务器只需要维持一个私钥就能够和多个客户端进行加密通信，但服务器发出的信息能够被所有的客户端解密，且该算法的计算复杂，加密速度慢。</p>

<ul>
<li>对数据进行加密，并建立一个信息安全通道，来保证传输过程中的数据安全</li>
<li>对网站服务器进行真实身份认证</li>
